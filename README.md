# ContestRedbook

[韩文版 README](./README.ko.md)

一个把“比赛信息整理”直接做成“小红书可发内容”的前端小工具。

它不是传统意义上的管理后台，也不是复杂的 SaaS，而是一个非常直接、非常实用的内容生产器：输入比赛信息，页面就会自动生成 4 张适合社媒发布的帖子图片，同时附带文案和文字版信息；如果你愿意再往前走一步，它还支持从比赛官网抓取内容，并借助 DeepSeek 自动完成信息抽取、风格推荐和视觉主题生成。

从结果看，这是一个“比赛资讯包装器”；从实现看，它其实是一个相当完整的单文件 Web App。

---

## 这个项目在做什么

`ContestRedbook` 解决的是一个很具体、也很真实的问题：

很多比赛信息明明散落在官网里，但真正要把它们整理成一篇适合小红书发布的内容，往往要经历下面这套重复劳动：

1. 手动读官网
2. 提炼比赛名称、时间、资格、奖项、主题
3. 组织成适合传播的文案结构
4. 再做成封面图、信息图、风格推荐图、奖项图
5. 最后导出成可发布的图片

这个项目把这条链路几乎打通了。

用户在页面里填写或导入比赛信息后，可以一次性得到：

1. 封面图
2. 比赛信息卡
3. 风格方向推荐卡
4. 奖项详情卡
5. 一份可复制的小红书文案
6. 一份纯文字版比赛总结

它面向的不是“所有互联网用户”，而是非常明确的内容工作流场景：比赛资讯整理、设计竞赛解读、社媒内容包装、设计类账号运营。

## 我对这个仓库的第一印象

这个仓库最有意思的地方，是它非常“轻”。

整个项目核心几乎都在一个 [`index.html`](/Users/ikkyux/Documents/Codex/2026-05-26/ikkyux-contestredbook-https-github-com-ikkyux/repo/index.html) 里完成，包含：

1. 页面结构
2. 视觉样式
3. 表单交互
4. AI 请求逻辑
5. 数据填充逻辑
6. 帖子模板生成
7. PNG 导出能力

这意味着它没有打包流程、没有框架依赖、没有构建门槛，打开就能跑。

这种写法在工程化语境里不算“主流最佳实践”，但对这类个人效率工具来说，它有一个非常强的优点：**交付快、迁移轻、理解成本低**。

如果你的目标是做一个能马上用起来的工具，而不是做一个可扩展十年的平台，这种方案其实很有力量。

## 核心能力拆解

### 1. 表单驱动的比赛信息录入

页面首先提供了一套比较完整的比赛信息表单，覆盖：

1. 中英文比赛名
2. 主办方
3. 参赛资格
4. 比赛主题与 Slogan
5. 截止日期、公布日期
6. 提交内容
7. 关键词
8. 含金量评分
9. 主题解析
10. 风格方向
11. 奖项与福利

这一步很关键，因为它把原本松散的比赛资料，先压成了一套结构化数据模型。后面所有图片和文案，其实都是基于这套数据模型二次生成。

### 2. 从官网抓取并用 AI 自动填表

项目并不满足于“手动输入”，而是额外做了一条自动化链路：

1. 输入比赛官网链接
2. 优先直接抓官网内容
3. 如果跨域或站点限制失败，则走 `r.jina.ai` Reader 转 Markdown
4. 把抓到的正文喂给 DeepSeek
5. 要求模型返回结构化 JSON
6. 自动回填到页面表单

这一段设计很实用，因为它没有试图做复杂爬虫，而是采用“网页读取 + LLM 结构化解析”的轻量策略。对非标准化官网来说，这种方式往往比写一堆站点特化规则更省事。

### 3. AI 生成风格方向

项目不只抽取事实信息，还进一步生成“内容观点”。

它要求模型基于比赛主题，生成 4 个差异明显的参赛风格方向，并强制覆盖这些维度：

1. 先锋实验
2. 文化叙事
3. 自然 / 材料
4. 商业佩戴或互动体验

这一步让工具从“信息整理器”升级成了“内容策划助手”。它不只是告诉你比赛是什么，还开始告诉你“可以怎么讲”“可以怎么创作”。

### 4. AI 生成视觉主题

除了文案层面的辅助，项目还把 AI 用到了视觉系统上：

1. 生成主色、辅色、深浅背景色
2. 选择版式布局
3. 选择封面模板
4. 尽量避免重复当前模板

这说明作者考虑的不只是“有没有图”，而是“图能不能持续换风格”。对于需要高频产出的内容账号来说，这个点很重要。

### 5. 四张帖子图的模板化输出

项目最终产出的是 4 张完整的社媒图片：

1. 封面图：负责吸引点击
2. 信息卡：负责传递关键赛事实用信息
3. 风格推荐卡：负责输出观点与启发
4. 奖项详情卡：负责强化参赛价值感

这套切分很像成熟内容账号的发布结构，不是单纯“把信息贴上去”，而是按照传播逻辑做了内容分工。

### 6. 一键导出 PNG

项目通过 `html2canvas` 把卡片 DOM 直接转成图片，并支持：

1. 下载当前卡片
2. 一次性下载全部卡片

这让整个工具闭环了。用户不需要再开设计软件二次截图，生成即交付。

## 这个项目为什么值得写进作品集

我觉得它的价值不在“技术栈多新”，而在于它体现了一种很清晰的产品判断：

### 1. 选题很小，但需求非常真实

很多个人项目失败，不是因为做不出来，而是因为问题太空。

这个项目反过来，它解决的是一个非常窄、但确实存在的工作流问题：把比赛官网内容快速转成可发布的小红书帖子。

问题越具体，产品越容易成立。

### 2. 它把 AI 放在了合适的位置

这里的 AI 不是“为了 AI 而 AI”，而是分别承担了三种非常合理的角色：

1. 信息抽取器
2. 风格策划助手
3. 视觉方案生成器

也就是说，AI 不负责整个系统，只负责人最嫌麻烦、最耗时间、又最适合结构化生成的部分。这是很成熟的使用方式。

### 3. 它是结果导向的

很多工具停留在“帮你整理信息”这一步，但实际内容工作者真正想要的是“可发的成品”。

这个项目从输入一路走到输出图片和文案，说明作者更关注交付结果，而不是中间过程炫技。

## 实现思路分析

如果把这个项目当作一个小型内容生成系统，它的内部链路大概是这样的：

```text
比赛官网 / 手动输入
        ↓
结构化表单数据
        ↓
DeepSeek 抽取 / 生成
        ↓
统一的数据字段
        ↓
四张卡片模板渲染
        ↓
文案生成 + 文字版摘要
        ↓
html2canvas 导出 PNG
```

它的实现特点主要有三点。

### 1. 单文件架构

所有逻辑都放在一个 HTML 文件中，典型特点是：

1. 上手快
2. 发布简单
3. 修改直观
4. 适合原型和个人工具

代价则是：

1. 代码体积会持续膨胀
2. 样式、模板、逻辑耦合较高
3. 后续多人协作不方便

但从当前仓库体量和目标看，这个取舍是成立的。

### 2. 数据先行，而不是模板先行

虽然表面上这是个“出图工具”，但代码结构上并不是先写死卡片内容，而是先收集一组统一字段，再让不同卡片消费这些字段。

这说明作者已经具备基本的“内容模型”意识，而不是单纯拼页面。

### 3. 视觉系统做了可变模板层

项目不仅写了一种封面，而是准备了多套视觉模板和布局变体，比如：

1. `crystal`
2. `cyber`
3. `ink`
4. `botanical`
5. `atelier`
6. `poster`
7. `split`
8. `blueprint`
9. `brutal`
10. `luxe`
11. `neon`
12. `catalog`

这部分很能体现作者的产品直觉。因为社媒内容如果长期只用一个壳，很快会显得重复；而模板层一旦抽出来，工具就具备了持续“换皮”的能力。

## 目前的局限

如果把这当作一个作品，我会觉得它已经很好用；如果把它当作一个长期演进的产品，下面这些点会是下一阶段值得处理的问题。

### 1. 代码耦合度偏高

HTML、CSS、模板、AI 调用、数据映射、导出逻辑都在同一文件里。短期开发效率高，长期维护成本会慢慢上升。

### 2. 强依赖第三方接口

AI 自动填表、风格生成、视觉主题生成都依赖 DeepSeek；官网抓取部分又依赖浏览器可访问目标站点，或 `r.jina.ai` 的可用性。

这意味着工具可用性会受到外部服务波动影响。

### 3. 领域适配目前比较垂直

从默认文案、风格提示、字段设计来看，它更偏向设计类比赛，尤其是珠宝设计比赛。若要扩展到建筑、插画、摄影、科技赛事，可能需要额外的领域模板。

### 4. 缺少更正式的配置与持久化

目前 API Key 存在浏览器页面上下文中，表单数据也没有看到持久化方案。对于频繁使用者来说，后续可以加入本地存储或导入导出配置能力。

### 5. 工程化空间仍然很大

如果未来要继续扩展，我会建议逐步拆分：

1. `data schema`
2. `prompt builder`
3. `template renderer`
4. `export module`
5. `theme system`

这样后面无论迁移到 Vue、React，还是继续原生维护，都会轻松很多。

## 如何运行

这个项目没有复杂依赖，最简单的方式就是直接打开 [`index.html`](/Users/ikkyux/Documents/Codex/2026-05-26/ikkyux-contestredbook-https-github-com-ikkyux/repo/index.html)。


然后访问：

```text
http://localhost:8000
```

## 我会怎么继续迭代它

如果把这个项目继续往前推进，我会优先做这几件事：

1. 把 `index.html` 拆成 `HTML / CSS / JS` 三层，先降低维护成本
2. 增加 `localStorage`，保存最近一次表单内容和 API 配置
3. 把“比赛类型”做成可选模板，比如珠宝、建筑、摄影、插画
4. 提供可编辑的 Prompt 面板，方便高级用户自己微调生成风格
5. 增加封面比例选项，支持小红书之外的平台
6. 补一份示例截图，让第一次访问仓库的人立刻知道它生成什么
